# CLAUDE.md — Passômetro (medishift-planner)

Guidance for AI assistants working on this repository.

## Project Overview

**Passômetro** is a client-side healthcare management web application for Hospital Neurocardio's ICU (UTI Geral). It manages patient tracking, antibiotic schedules, and shift assignments. The backend runs on Azure Web App at `passometro.azurewebsites.net`.

This repository is a **bug-fix branch** — it contains targeted corrections to the production `medishift-planner.js` file.

## Repository Structure

## Runtime Configuration

The API base URLs are configured via injected globals (Azure Web App pattern):

```js
const API_URL      = window.__API_URL__      || '/api/patients';
const SCHEDULE_API = window.__SCHEDULE_API__ || '/api/schedule';
```

No `.env` files — configuration is injected by the host before the script loads.

## API Contracts

### `/api/patients`

```
GET  /api/patients
     → { success: true, patients: [...] }

POST /api/patients   { action: 'savePatient', patient: { id, name, bedNumber, ... } }
     → { success: true }

POST /api/patients   { action: 'deletePatient', id: 'pat_...' }
     → { success: true }   ← KNOWN BUG: backend ignores this, see below
```

### `/api/schedule`

```
GET  /api/schedule?month=YYYY-MM
     → { success: true, entries: [...] }

POST /api/schedule   { entry: { id, assignedTo, swapTo, ... } }
     → { success: true }
```

## Data Shapes

**Patient object:**
```js
{
  id: 'pat_TIMESTAMP_randomString',
  name: string,
  bedNumber: string,
  admissionDate: string,   // ISO date
  priority: 'Alta' | 'Média' | 'Baixa',
  diagnosis: string,
  author: string,
  antibiotics: [{ name: string, startDate: string }]  // normalized form
}
```

**Antibiotic data** arrives in multiple formats from the backend (string, JSON string, array). `normalizeAntibiotics()` handles all cases.

**Schedule entry:**
```js
{ id: 'DD-MM__shiftCode', assignedTo: string, swapTo: string | null }
```

## Known Bugs

### Bug 1 — FIXED in this branch
**Symptom:** "Paciente atualizado!" alert always appeared, even for new patients.  
**Root cause:** `patientData.id` was always truthy (ID was generated before the condition check).  
**Fix:** Check the hidden `#patientId` form field *before* generating an ID; pass an `isNew` flag to `savePatient()`.  
**Location:** `handleFormSubmit()` around line 386–410.

### Bug 2 — NOT FIXED (backend issue)
**Symptom:** Delete button does not delete patients.  
**Root cause:** The backend API handler ignores `action: 'deletePatient'` and processes the request as a save, blanking fields instead.  
**Fix required:** Backend `api.js` needs to handle `action:'deletePatient'` separately, or expose a `DELETE /api/patients/:id` endpoint.

## Key Conventions

### Naming
- Functions: `camelCase` — e.g., `calculateDIH`, `normalizeAntibiotics`, `renderPatients`
- Patient IDs: `pat_TIMESTAMP_randomHex` — generated by `generateId()`
- Shift IDs: `DD-MM__shiftCode` — e.g., `05-04__G1`
- CSS classes: `kebab-case` — e.g., `patient-card`, `priority-high`, `antibiotic-days`
- HTML element IDs: descriptive, match their logical name — e.g., `#patientId`, `#patientsGrid`

### Code Style
- No module system — all functions are global
- DOM access via `getElementById()` and `querySelector()` by ID
- Dynamic HTML via template literals and `innerHTML` assignment
- `escapeHtml()` must be used for all user-supplied content inserted into the DOM
- Dates are ISO strings; relative display is handled by `formatDateTime()`
- All user-facing strings and labels are in **Portuguese**

### Priority Mapping
```js
'Alta'  → CSS class 'high'
'Média' → CSS class 'medium'
'Baixa' → CSS class 'low'
```

### Shift Types
- Weekdays: Diarista G1 (08–16), Vespertino (14–20), Noturno (20–08)
- Weekends: Diurno (08–20), Noturno (20–08)
- Fixed rule: Janio Euler is always Diarista G1 on weekdays (hardcoded)

## Development Workflow

Since there is no test suite, validate changes by:

1. Reading the section you intend to modify and understanding its full context.
2. Tracing all callers of any function you change.
3. Checking that `escapeHtml()` is applied to any new user-supplied strings rendered into HTML.
4. Verifying antibiotic-related changes work with all three input formats (array, JSON string, plain string).
5. Confirming modal open/close state is properly reset in `closeModal()` after any form changes.
